# Аудитор кода

Аудитор кода по ошибке из журнала мониторинга находит **упавший метод**, его **граф вызовов**
и ставит **диагноз через LLM**, записывая результат обратно в 1С. Вместо «простыни» модуля в
модель уходит окно: метод + вызывающие его места.

## Как устроено

```
1С (КИИ_АудиторКода) ──{модуль, строка}──▶ ШИМ (REST) ──▶ rlm (индекс кода) ──▶ метод + граф
                     ◀─── диагноз в журнал ─── LLM ◀── КИИ_КоннекторИИ ◀──────────┘
```

Три принципа, которые важно понимать при настройке:

1. **Механика — снаружи.** Разбор кода делает внешний движок `rlm-tools-bsl` (в BSL это
   невозможно). Он stateless — держит только индекс (копию исходников), сносится без потерь.
2. **Ум — через коннектор.** Диагноз ставит LLM, и обращение к модели идёт **только** через
   `КИИ_КоннекторИИ`. Поэтому модель настраивается централизованно, а не в аудиторе.
3. **Отчёт — внутрь.** Результат падает в 1С (журнал мониторинга, дашборд). На внешнем движке
   ничего ценного не оседает.

Между 1С и движком стоит тонкий **REST-шим** (входит в поставку аудитора): он прячет сложность
MCP-протокола, 1С шлёт простой JSON и получает готовый ответ.

## Что нужно

- Развёрнутый движок: **шим** + **rlm-tools-bsl** (см. шаг 1).
- Настроенный `КИИ_КоннекторИИ` с доступом хотя бы к одной модели (см. «Настройка моделей»).
- XML-выгрузка конфигурации, которую аудитор будет анализировать (см. шаг 2).
- Право на чтение кода включается отдельным флагом (по умолчанию **выключено**, см. «Безопасность»).

## Шаг 1. Развернуть движок (шим + rlm)

Движок можно развернуть тремя способами — выберите по инфраструктуре. Полные детали и файлы —
в каталоге поставки аудитора (`DEPLOY.md`, `deploy/`).

### Вариант A — Windows-служба (файловая база или одиночный Windows-сервер)

Самый надёжный автозапуск на Windows. Оба компонента — как службы, автостарт.

```powershell
# 1) rlm — родным установщиком (PowerShell от администратора):
irm https://raw.githubusercontent.com/Dach-Coin/rlm-tools-bsl/master/simple-install-from-pip.ps1 -OutFile simple-install-from-pip.ps1
PowerShell -ExecutionPolicy Bypass -File .\simple-install-from-pip.ps1

# 2) шим — из каталога поставки (PowerShell от администратора):
PowerShell -ExecutionPolicy Bypass -File .\deploy\windows\install-shim-service.ps1
```

### Вариант B — Docker (оптимально на Linux-хосте)

```bash
cd deploy/docker && cp .env.example .env
# заполните пути к движку rlm и к XML-дампу конфигурации
docker compose up -d --build
```

Поднимаются два сервиса: `rlm` (внутренний) и `shim` (публикуется наружу, порт по умолчанию `9001`).

> На Windows Docker Desktop **не рекомендуется**: ввод-вывод исходников идёт через WSL2 и медленнее
> в разы. На Windows берите вариант A, Docker — для Linux-хоста.

### Вариант C — Linux без Docker (systemd)

rlm — родным `simple-install-from-pip.sh` (+ `loginctl enable-linger`), шим — unit-файлом из
`deploy/linux/`. Детали — в `DEPLOY.md`.

### Где размещать и как закрывать

| Топология | Где движок | Слушать | Firewall |
|---|---|---|---|
| Файловая база | тот же ПК, что и клиент 1С | `127.0.0.1` | не нужно |
| Серверная, 1 сервер | тот же хост, что и кластер | `127.0.0.1` | не нужно |
| Движок на отдельном хосте / кластер | отдельная машина | `0.0.0.0` / IP | **обязательно** |

> ⚠️ **Шим неаутентифицирован** — это REST, отдающий исходный код по координате. На `127.0.0.1`
> безопасно; при работе по сети закройте порт шима firewall'ом, пустив только адреса серверов 1С.

## Шаг 2. Подготовить индекс кода

rlm анализирует **XML-выгрузку** конфигурации (не живую базу). Один раз готовим дамп и индекс,
дальше обновляем при изменении кода.

1. Выгрузить конфигурацию в XML (Конфигуратор / `ibcmd` / пакетный режим), с расширениями.
2. Построить индекс:
   ```bash
   rlm-bsl-index index build <путь-к-дампу>
   # в Docker: docker compose exec rlm rlm-bsl-index index build /repos/<имя>
   ```

### Свежесть индекса — реиндекс на деплое

Индекс должен **совпадать с задеплоенным кодом**, иначе номера строк из стека разъедутся с телом
метода и диагноз соврёт. Поэтому реиндекс делается **на деплое в прод**, не по таймеру:

1. На деплое выгрузить конфигурацию в XML (тем же способом, что и при обновлении).
2. Обновить индекс инкрементально:
   ```bash
   rlm-bsl-index index update <путь-к-дампу>
   ```

Шаг удобно встроить в свой процесс деплоя (скрипт обновления / CI).

## Шаг 3. Настроить подсистему в 1С

Настройки — в базе (константы; при наличии — через форму настройки анализатора).

| Что | Тип | Значение |
|---|---|---|
| `КИИ_АдресХостаШимаАудитора` | Строка | пусто = авто: файловая → `127.0.0.1`, серверная → хост кластера. Явный адрес — если движок на отдельном хосте |
| `КИИ_ПортШимаАудитора` | Число | `9001` (или свой порт шима) |
| `МодельДляАнализа` | — | модель для диагноза (см. «Настройка моделей»). Если модель ходит через прокси — включите прокси у модели |
| `РазрешитьЧтениеКода` | Булево | `Истина`, чтобы аудитор мог читать код (по умолчанию **Ложь**) |

**Автозапуск (автономный агент).** Регламентное задание `КИИ_АудиторКодаАвтоанализ` (по умолчанию
раз в час) выбирает новые ошибки журнала, диагностирует и шлёт результат. Проверьте, что у задания
`Использование = Истина`, а сеанс, где крутятся регламенты, запущен с разрешением их выполнения.

## Шаг 4. Проверка

Из 1С:

- `КИИ_АудиторКода.ШимДоступен()` → `Истина`, если движок отвечает.
- `КИИ_АудиторКода.ПроанализироватьМестоОшибки(ИмяМодуля, НомерСтроки)` → метод, тело, граф вызовов.

Напрямую (замените адрес/порт):

```bash
curl http://<хост-движка>:9001/health
curl -X POST http://<хост-движка>:9001/audit -H "Content-Type: application/json" \
  --data-binary '{"module":"<ИмяМодуля>","line":<номер>}'
```

Ожидаемо возвращается имя метода, диапазон строк, тело и список вызывающих мест.

## Безопасность

- **Код ≠ данные.** Чтение кода — отдельное разрешение `РазрешитьЧтениеКода` (по умолчанию
  выключено, коробочный fail-closed). Включайте осознанно.
- **В модель уходит окно, не вся база** — метод плюс несколько вызывающих мест, а не конфигурация целиком.
- **Шим неаутентифицирован.** По сети закрывайте порт шима firewall'ом (только адреса серверов 1С).
  На `127.0.0.1` дополнительной защиты не требуется.
- **Диагноз ставит только коннектор.** Мимо `КИИ_КоннекторИИ` в модель обращений нет — весь доступ
  к LLM проходит через gateway и его аудит.

## Диагностика

| Симптом | Причина / решение |
|---|---|
| `ШимДоступен()` = Ложь, автоанализ даёт 0 диагнозов | Движок не запущен или недоступен. Проверьте службу/контейнер и `curl .../health`. Аудитор при недоступном шиме не падает — просто ничего не делает (fail-closed) |
| Диагноз не ставится, ошибки модели `resolve-host` / `403` | Модель ходит напрямую там, где нужен прокси. Настройте прокси у модели в коннекторе |
| Windows-служба rlm не стартует (ошибка `1053`) | Запустите диагностический скрипт из поставки rlm (`diagnose-service-win.ps1`) |
| Первый анализ очень долгий | Индекс не построен — rlm разбирает файлы на лету. Постройте индекс (шаг 2) |
| Диагноз ссылается не на те строки | Индекс устарел относительно кода. Обновите индекс (реиндекс на деплое) |

---

_Аудитор кода — приложение-витрина над ядром ИИконы (коннектор к моделям). Диагноз работает только
через коннектор: без ядра витрина не функционирует._
